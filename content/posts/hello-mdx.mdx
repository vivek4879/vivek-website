This site's blog is now powered by MDX files in the repo. Every post lives in
`content/posts/` as a `.mdx` file with a YAML frontmatter block at the top.

## Why MDX

Plain Markdown is enough for prose, but a developer blog eventually wants
**embedded React components** — interactive demos, custom callouts, charts.
MDX gives us that escape hatch without rebuilding the renderer later.

## What this post tests

This post exists mostly to verify the rendering pipeline. It exercises:

- Frontmatter parsing (`gray-matter`)
- GitHub-Flavored Markdown via `remark-gfm` (this list, tables, autolinks)
- Syntax highlighting via [`rehype-pretty-code`](https://rehype-pretty.pages.dev/)
- The full MDX → React compile path through `next-mdx-remote`
